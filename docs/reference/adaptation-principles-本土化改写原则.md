# 本土化改写原则（Adaptation Principles）

这份说明回答一个核心问题：

我们吸收别人的方法论时，到底是在抄术语，还是在拿走真正有用的骨架。

答案应该是后者。

## 我们到底在吸收什么

像王自如那类实践，或者 `gstack` 这种项目，真正值得吸收的不是它们的表层表达，而是下面这些稳定内核：

- 先把模糊需求收敛清楚，再进入实现
- 不让 AI 每次都从全仓扫描开始
- 把一次任务限制在清晰范围内
- 出问题先收证据，再动手修
- 推理一旦转向，要留下轨迹
- 修好的 bug 要沉淀成回归资产

这些东西换成英文说、换成中文说，本质都不变。

## 哪些东西不要照搬

- 不照搬别人的语气
- 不照搬看起来很酷但不适合你的命名
- 不照搬只适合英语母语者的表达习惯
- 不照搬和你当前开发能力、项目规模不匹配的重流程

表层形式可以改，核心机制不能丢。

## 中文化不是直译

直译最大的问题，不是“难看”，而是会让你误以为自己理解了。

比如：

- `task-packet` 不是简单翻成“任务数据包”，它在这里真正指的是“一次任务的工作单和边界说明”
- `debug-pack` 不是“调试包”，而是“排障前先整理好的证据包”
- `decision-trace` 不是抽象概念，而是“为什么改动方向变了”的记录

所以这里的中文化原则是：

- 先理解动作，再命名
- 先贴合使用场景，再考虑术语优雅
- 中文做主表达，英文做辅助别名

## 我们的写法标准

在这个仓库里，优先采用下面这种表达方式：

- 中文主名 + 英文别名
- 先说动作，再说术语
- 先让人能执行，再让人记住词

例如：

- `需求访谈（discovery interview）`
- `需求简报（requirement-brief）`
- `仓库地图（repo-map）`
- `排障包（debug-pack）`

而不是反过来把英文放在前面，让中文去做括号解释。

## 贴合你的实际意味着什么

这套协议不是给抽象团队写的，而是给你这种真实使用场景写的：

- 中文是主工作语言
- 英文术语需要逐步熟悉，但不能成为门槛
- 不想被流程绑死，但又确实需要边界和秩序
- 希望 AI 帮你提速，但不希望它一上来就把仓库看爆

所以我们追求的不是“翻译正确”，而是：

- 看得懂
- 用得上
- 不容易跑偏
- 能随着经验继续迭代

## 判断一次改写是否合格

如果一段话改完以后，满足下面这些点，就算合格：

- 你不查词典也知道下一步要做什么
- 这个词跟实际动作绑定得上
- 英文术语还在，但不会压住中文理解
- 你能把这套说法复述给别人

达不到这些，就说明还停留在“翻译层”，没有进入“工作语言层”。
