# 阶段路由（Stage Router）

这份手册回答一个简单但关键的问题：

现在这个任务，到底应该走到哪一阶段。

很多 AI 开发偏航，不是因为模型太笨，而是因为所有事情都被塞进了“直接实现”这一阶段。

## 核心原则

- 不是每个任务都应该从写代码开始
- 每个阶段只解决一类问题
- 上一阶段的输出，要成为下一阶段的输入
- 如果阶段问题还没解决，就不要硬跳到后面

## 标准主线

1. `需求访谈（discovery interview）`
2. `需求简报（requirement-brief）`
3. `产品评审（product review）`
4. `设计令牌包（design-token-pack）`，仅 UI / 多端任务推荐
5. `仓库地图（repo-map）`
6. `工程评审（engineering review）`
7. 必要时补 `领域地图 / 工具契约 / 执行计划 / 可观测性关联 / 成本权限预算`
8. `任务包（task-packet）` 或 `排障包（debug-pack）`
9. `根因调查（investigate）`，仅 bug 必过
10. 实现
11. `真实验收（live QA）`
12. `回归用例（eval-case）`，真实 bug 必补
13. `文档同步包（doc-sync）`

## 功能开发路线

适用：

- 新功能
- 增强
- 范围可控的重构

路线：

1. 需求访谈
2. 需求简报
3. 产品评审
4. UI 任务先补设计令牌包
5. 仓库地图
6. 工程评审
7. 多业务域任务补领域地图与工具契约
8. 长流程任务补执行计划
9. 任务包
10. 实现
11. 真实验收
12. 文档同步

## Bug 修复路线

适用：

- 用户反馈 bug
- 测试失败
- 线上行为和预期不一致

路线：

1. 仓库地图
2. 排障包
3. 可观测性关联
4. 根因调查
5. 定点修复
6. 真实验收
7. 回归用例
8. 文档同步

## 阶段门

下面这些门没有过，就不要往后走：

- 需求没收敛，不进产品评审
- 产品评审没定 scope，不进工程评审
- UI 任务如果视觉语言还没定，不要直接生成组件
- 多业务域任务如果边界和工具语义没定，不要直接开任务包
- 工程评审没定边界，不开任务包
- 没有排障包和根因调查，不修 bug
- 没有真实验收，不算完成
- 没有文档同步，不算正式收尾

## 判断当前卡在哪一层

如果你在问下面这些问题，说明你还不该直接开写：

- “我到底在解决谁的问题？”
- “第一版到底做多大？”
- “这一层状态应该归谁管？”
- “这个 bug 到底是 UI 问题、状态问题，还是标识问题？”
- “单元测试过了，真实流程到底通没通？”
- “这次改动只是补一段 README，还是其实整个介绍都过期了？”

## 路由的真正价值

阶段路由不是为了把流程变重，而是为了让每个阶段只承担自己该承担的复杂度。

你越早把问题放对阶段，后面的 token、返工和偏航就越少。

现在产品评审和工程评审都已经有结构化工件：

- `product-review`
- `engineering-review`

如果已经有对应输入，优先生成工件，再进入后续 `task-packet` 或 `debug-pack`。
