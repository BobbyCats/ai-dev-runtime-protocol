# 阶段路由（Stage Router）

这份手册回答一个简单但关键的问题：

现在这个任务，到底应该走到哪一阶段。

很多 AI 开发偏航，不是因为模型太笨，而是因为所有事情都被塞进了“直接实现”这一阶段。

## 核心原则

- 不是每个任务都应该从写代码开始
- 每个阶段只解决一类问题
- 上一阶段的输出，要成为下一阶段的输入
- 如果阶段问题还没解决，就不要硬跳到后面

## 标准主线

1. `需求访谈（discovery interview）`
2. `需求简报（requirement-brief）`
3. `产品评审（product review）`
4. `工程评审（engineering review）`
5. `仓库地图（repo-map）`
6. `任务包（task-packet）` 或 `排障包（debug-pack）`
7. `根因调查（investigate）`，仅 bug 必过
8. 实现
9. `真实验收（live QA）`
10. `回归用例（eval-case）`，真实 bug 必补
11. `文档同步包（doc-sync）`

## 功能开发路线

适用：

- 新功能
- 增强
- 范围可控的重构

路线：

1. 需求访谈
2. 需求简报
3. 产品评审
4. 工程评审
5. 仓库地图
6. 任务包
7. 实现
8. 真实验收
9. 文档同步

## Bug 修复路线

适用：

- 用户反馈 bug
- 测试失败
- 线上行为和预期不一致

路线：

1. 仓库地图
2. 排障包
3. 根因调查
4. 定点修复
5. 真实验收
6. 回归用例
7. 文档同步

## 阶段门

下面这些门没有过，就不要往后走：

- 需求没收敛，不进产品评审
- 产品评审没定 scope，不进工程评审
- 工程评审没定边界，不开任务包
- 没有排障包和根因调查，不修 bug
- 没有真实验收，不算完成
- 没有文档同步，不算正式收尾

## 判断当前卡在哪一层

如果你在问下面这些问题，说明你还不该直接开写：

- “我到底在解决谁的问题？”
- “第一版到底做多大？”
- “这一层状态应该归谁管？”
- “这个 bug 到底是 UI 问题、状态问题，还是标识问题？”
- “单元测试过了，真实流程到底通没通？”
- “这次改动只是补一段 README，还是其实整个介绍都过期了？”

## 路由的真正价值

阶段路由不是为了把流程变重，而是为了让每个阶段只承担自己该承担的复杂度。

你越早把问题放对阶段，后面的 token、返工和偏航就越少。
