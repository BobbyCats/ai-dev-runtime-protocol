# Feature Playbook | 功能开发手册

适用场景：

- 新功能
- 小范围增强
- 可控范围内的重构

## 标准顺序

1. 需求还模糊时，先做 `需求访谈（discovery interview）`
2. 落成 `需求简报（requirement-brief）`
3. 先过 `产品评审（product review）`
4. 如果是 UI / 多端任务，先补 `设计令牌包（design-token-pack）`
5. 再过 `工程评审（engineering review）`
6. 生成或刷新 `仓库地图（repo-map）`
7. 再写 `任务包（task-packet）`
8. 只读任务包给出的候选文件
9. 实现过程中如果权衡改变，写入 `决策轨迹（decision-trace）`
10. 跑验证命令
11. 走 `真实验收（live QA）`
12. 如果引入或顺手发现了真实 bug，把它补成回归用例
13. 收尾前生成 `文档同步包（doc-sync）`

## 硬规则

- 范围必须落在任务包里，不要散在聊天记录里
- 没过产品评审和工程评审，不要直接开工
- UI 任务没过设计令牌这关，不要直接生成组件
- 没写验收标准，不要直接实现
- 候选文件一旦超出上下文预算，应该拆任务而不是硬顶
- 一旦跨域，重新写任务包
- 影响真实用户路径的功能，没有真实验收就不算完成
- 如果命令、工作流或定位变了，README 不能只补一条，必须走文档同步判断
