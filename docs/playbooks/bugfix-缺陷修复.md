# Bugfix Playbook | 缺陷修复手册

适用场景：

- 用户反馈 bug
- 测试突然失败
- 线上行为和预期不一致

## 标准顺序

1. 先生成或刷新 `仓库地图（repo-map）`
2. 再生成 `排障包（debug-pack）`
3. 先复现，再做 `根因调查（investigate）`
4. 先读排障包里的短名单，不要一上来全仓扫描
5. 关键判断变化时，写入 `决策轨迹（decision-trace）`
6. 只修最能解释问题的那一层
7. 跑验证命令
8. 走 `真实验收（live QA）`
9. 把这个 bug 变成 `回归用例（eval-case）`
10. 收尾前生成 `文档同步包（doc-sync）`

## 硬规则

- 没有先复现，就不要大范围改逻辑
- 没有排障包，就不要默认全仓搜索
- 没有根因调查，不要直接开修
- 没有证据，不要凭感觉扩大范围
- 对真实用户有影响的 bug，修完必须留下回归资产
- 影响真实入口的 bug，没有真实验收就不算关闭
- 如果这次改动改变了命令、工作流或系统定位，README 需要按章节或整篇重写

## 最少需要的工件

- `repo-map`
- `debug-pack`
- `decision-trace`
- `eval-case`
- `doc-sync`
