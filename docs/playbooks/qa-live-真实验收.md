# 真实验收（Live QA）

适用场景：

- 真实用户会碰到的页面、命令、接口、文件输出发生变化
- 你已经写了代码，也跑了测试，但还没从真实入口走一遍

## 这一步的目标

不是多跑几条测试，而是确认：

- 真实入口能不能走通
- 真实界面或真实输出是不是符合预期
- 关键边界场景有没有被验证到

## 应该验什么

- Web / App 页面：真实点击、真实输入、真实结果
- CLI：真实命令、真实参数、真实输出
- API：真实请求、真实响应、真实错误分支
- 文件产物：真实生成、真实格式、真实可读性

## 标准顺序

1. 明确要验的真实入口
2. 跑 happy path
3. 跑最容易翻车的边界场景
4. 记录证据
5. 给出通过 / 不通过结论

推荐模板：

- [templates/qa-report-真实验收报告.md](../../templates/qa-report-真实验收报告.md)

## 硬规则

- 单元测试不能替代真实验收
- 没有真实证据，不要口头宣布“应该没问题”
- QA 发现问题时，回到 `排障包（debug-pack）` 和 `根因调查（investigate）`
- 对真实用户路径有影响的改动，没有真实验收就不算完成
