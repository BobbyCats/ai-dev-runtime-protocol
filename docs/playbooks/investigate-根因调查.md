# 根因调查（Investigate）

适用场景：

- 用户报告 bug
- 测试失败，但不知道真正原因
- 你已经有修法冲动，但还没有足够证据

## 这一步的目标

不是“尽快修一下”，而是先找到最能解释问题的最小原因。

## 标准顺序

1. 先复现
2. 先看 `排障包（debug-pack）`
3. 先按关联编号聚焦日志与故障阶段
4. 列出 1 到 3 个最可能的假设
5. 用日志、trace、代码和真实行为去逐个排除
6. 写出一句清楚的根因陈述
7. 只修最能解释问题的那一层

## 根因陈述应该长什么样

好的根因陈述通常像这样：

- “删除动作使用了列表索引，而不是稳定事件 ID，所以 UI 在排序变化后会删错对象。”
- “缓存键缺了用户维度，导致不同用户请求互相污染。”

它应该能同时解释：

- 为什么会出 bug
- 为什么不是别的地方先坏
- 为什么你的修法正好能封住这个问题

## 硬规则

- 没复现，不要修
- 没先看关联编号与日志聚焦，不要盲目扩大代码搜索
- 没有能解释问题的根因陈述，不要修
- 连续 3 个假设都被证伪时，先质疑 framing，不要盲目扩大搜索
- 根因判断变化时，要补 `决策轨迹（decision-trace）`
- 修好以后，要把真实 bug 变成 `回归用例（eval-case）`
